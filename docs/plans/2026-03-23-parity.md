## Tier 1: Correctness -- The System Cannot Work Without These

### 1. System prompt is silently dropped
`streaming.py:162` -- The orchestrator receives the system prompt in `config` but
never passes it to `ChatRequest.system` or injects it as a system `Message`. Every
provider call goes out with no system prompt. The agent has no instructions.

### 2. stderr pipe never consumed -- services will deadlock
`lifecycle.py:53` -- Service subprocesses have `stderr=PIPE` but nobody reads it.
Once a service writes >64KB to stderr (tracebacks, logging), the pipe buffer fills
and the service blocks on write, stalling all IPC.

### 3. Spawn depth limit is dead code -- infinite recursion possible
`host.py:365-373` -- `_build_spawn_handler` never passes `current_depth` to
`spawn_child_session`. Every child spawn starts at depth 0. Max depth (3) is
unreachable. An agent that delegates to itself will fork-bomb.

### 4. Hook results overwritten -- only last hook's result survives
`server.py:548-556` -- The hook dispatch loop assigns `final_result = result` on
every iteration. If Hook 1 returns `INJECT_CONTEXT` and Hook 2 returns `CONTINUE`,
the injection is lost.

### 5. `ASK_USER` hook action ignored -- approval gates bypassed
`streaming.py:130-134, 301-315` -- The orchestrator checks for `DENY` but not
`ASK_USER`. Any hook returning `ASK_USER` (tool approval, safety gates) is treated
as `CONTINUE`.

### 6. `TOOL_PRE` MODIFY result discarded -- hooks can't rewrite tool args
`streaming.py:301-321` -- If a hook modifies tool arguments via `HookAction.MODIFY`,
the orchestrator ignores the returned data and always uses the original arguments.
Redaction hooks, path rewriting hooks, etc. are silently ineffective.

### 7. Hook fan-out is local-only -- external service hooks never called
`server.py:751-752` -- When the orchestrator emits hooks via `_OrchestratorLocalClient`,
only hooks in the same process are invoked. Hooks in external services (modes,
routing-matrix, skills) are never called from the orchestrator path.

### 8. Token estimation inflated 3-5x -- premature context compaction
`simple.py:1016-1018` -- `_estimate_tokens` uses `str(msg)` which produces the
Pydantic repr including field names and `None` values. A 10-char message estimates
as ~20 tokens. Compaction triggers far too early.

### 9. Context budget disconnected from actual model
`streaming.py:137` -- `context_get_messages` is called with empty `{}` params. The
context manager never receives the actual model's context window. It compacts against
a hardcoded 200k budget regardless of actual model.

## Tier 2: Reliability -- Hangs, Crashes, and Data Loss

### 10. No timeout on any request after startup
`client.py:88-106` -- `Client.request()` awaits a future indefinitely. If a service
crashes, the host/orchestrator hangs forever. Only `describe` and `configure` have
timeouts.

### 11. `LimitOverrunError` crashes server or host
`framing.py:19`, `server.py:222-229` -- Only `ValueError` is caught at read sites.
`asyncio.LimitOverrunError` propagates uncaught and kills the process.

### 12. Client future leaked on write failure
`client.py:102-106` -- If `write_message` raises, the future stays in `_pending`
forever.

### 13. Transcript load crashes on corrupted JSONL
`persistence.py:50-53` -- No `try/except` around `json.loads`. A partial line from
a crash makes the session un-resumable.

### 14. Non-atomic persistence writes -- data loss on crash
`persistence.py:29-32, 63-66` -- `state.json` and `metadata.json` use `open("w")`
(truncate-then-write). A crash between truncate and write loses the file.

### 15. Async generator teardown depends on GC
`host.py:145-288` -- `run()` is an async generator with cleanup in `finally`. If
the consumer drops the generator without `aclose()`, services may never be torn down.

### 16. IPC reader shared between handle_stream and Client._read_loop
`server.py:783-787` -- When the orchestrator makes IPC calls, the client's read loop
and the server's handle_stream can race on the same `StreamReader`.

### 17. Notification handler exception kills all pending requests
`client.py:67-72` -- A crash in `on_notification` causes `_fail_all_pending`.

## Tier 3: Feature Parity Gaps

### 18. OpenAI provider is a major regression
Old: 2335 lines with Responses API, deep_research, continuation tokens, retry.
New: 521 lines with basic Chat Completions only.

### 19. Delegate tool is skeletal
Old: 678 lines with provider preferences, tool/hook inheritance, self-delegation
depth tracking, context depth/scope. New: 116 lines, basic spawn/resume.

### 20. 4 hooks are stubs
`todo_reminder`, `session_naming`, `todo_display`, `streaming_ui` all return
`CONTINUE` as no-ops. Root cause: IPC hooks can't access shared session state or
make LLM calls.

### 21. No provider retry logic
Old system had `retry_with_backoff` + `PROVIDER_RETRY` events. New providers surface
`retryable` metadata but nobody retries.

### 22. 10 missing CLI slash commands
`/mode`, `/save`, `/status`, `/clear`, `/config`, `/tools`, `/agents`, `/rename`,
`/fork`, inline `/allowed-dirs`.

### 23. No orchestrator cancellation protocol
Old kernel had cooperative `CancellationToken` checked between tool calls. New
orchestrator has no way to abort mid-execution.

### 24. Missing execution:start/end events
Old orchestrator contract requires these. New doesn't emit them.

### 25. `prompt:submit` MODIFY not applied
Hooks that rewrite user prompts are silently ignored.
